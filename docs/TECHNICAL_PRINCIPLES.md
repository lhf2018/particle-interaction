# FluidVibe 技术实现原理

本文档说明本项目的核心技术路径与关键设计取舍。

## 1. 总体架构

项目由两条主线组成：

1. **本地高性能渲染链**（`fluid_sim_3d.py`）
   - 目标：高交互、高帧率、充分使用本机 GPU
2. **Web 预览链**（`app.py` + `web/`）
   - 目标：轻量访问与远程预览，非核心渲染入口

---

## 2. 本地 3D 程序原理（`fluid_sim_3d.py`）

### 2.1 数据结构

- 粒子位置：`pos (N, 3)`，`float32`，GPU 数组
- 粒子速度：`vel (N, 3)`，`float32`，GPU 数组
- 障碍体：当前版本已移除内部碰撞体，仅保留外层空间容器

### 2.2 物理更新流程

每帧步骤（GPU）：

1. 鼠标力场（吸引/排斥 + 旋转扰动）
2. 冲击波脉冲（点击触发）
3. 噪声扰动与底部对流
4. 重力与阻尼
5. 速度裁剪（限速）
6. 外层球形容器碰撞反射

### 2.3 GPU 优化策略

- 尽量使用 `cupy` 向量化，避免 Python 循环
- 减少 `cp.any()` 等同步点造成的 CPU-GPU 栅栏
- 渲染采用 **GPU 光栅化优先**：
  - 在 GPU 上完成投影/颜色/点写入
  - 每帧只回传最终画布，而非全量粒子中间态
- GPU 监控采样降频，降低 CPU 轮询开销

### 2.4 交互系统

- 左键：吸引
- 右键：排斥
- 中键：暂停/恢复（双击重置）
- 滚轮：调力
- UI 滑块：粒子数量目标值（拖动预览，松开生效）

---

## 3. Web 预览链原理（`app.py` + `web/main.js`）

### 3.1 通信模型

- 协议：WebSocket（双向）
- 前端 -> 后端：鼠标输入/参数变更
- 后端 -> 前端：粒子数据流（压缩二进制）

### 3.2 后端计算

- 默认优先 CUDA（`CuPy`）
- 不可用时回退 CPU（`NumPy`）
- 通过采样与量化控制带宽和抖动

### 3.3 数据压缩

为降低传输压力，后端将坐标量化为 `uint16`：

- Header: `count + sim_width + sim_height`
- Body: `uint16` 坐标流

前端按当前画布尺寸解码还原并渲染。

---

## 4. 关键性能瓶颈与处理

### 瓶颈 A：CPU-GPU 同步过多

- 现象：帧率抖动、交互滞后
- 处理：减少同步点，尽量在 GPU 完成整段计算

### 瓶颈 B：WebSocket 推流过大

- 现象：`keepalive ping failed`、前端卡顿
- 处理：降采样、降帧、二进制压缩、异常帧校验

### 瓶颈 C：浏览器渲染链不稳定

- 现象：白屏或掉帧
- 处理：上下文降级、渲染 try/catch、模式降级策略

---

## 5. 为什么“本地 3D”为核心

浏览器端受限于安全沙箱和渲染管线，不适合承担极限 CUDA 场景。  
因此项目定位为：

- **本地 3D = 核心体验**
- **Web = 预览与联调**
