# FluidVibe 技术实现原理

本文档说明本项目的核心技术路径与关键设计取舍。

## 1. 总体架构

项目由三条主线组成：

1. **本地高性能渲染链**（`fluid_sim_3d.py`）
   - 目标：高交互、高帧率、充分使用本机 GPU
2. **2D 黑洞–行星粒子模拟**（`black_hole_planet_2d.py`）
   - 目标：引力数值实验、离线可视化；与 Web 无关，界面截图见仓库根目录 **`img_2.png`**
3. **Web 预览链**（`app.py` + `web/`）
   - 目标：轻量访问与远程预览，非核心渲染入口

---

## 2. 2D 黑洞模拟原理（`black_hole_planet_2d.py`）

### 2.1 定位与产物

- **独立脚本**：仅依赖 Matplotlib（UI）与 NumPy/CuPy（数值），不经过 `app.py` 与 `web/`。
- **`img_2.png`**：用于文档展示的典型运行界面截图，与该脚本一一对应。

### 2.2 物理模型

- 中心黑洞对全体粒子施加 **软化牛顿引力**（避免 \(r\to 0\) 数值发散）。
- 可选 **行星粒子弱自引力**：存活粒子两两求和，复杂度 \(O(n_{\mathrm{alive}}^2)\)，默认在偶数子步计算以降低负载。
- **吞噬判定**：粒子与黑洞距离小于「视界半径」（与黑洞质量及可调比例系数相关）后从存活集合移除。

### 2.3 计算后端

- **优先 CuPy（CUDA）**：位置、速度、加速度使用 `float32` GPU 数组。
- **回退 NumPy**：无可用 CUDA 时自动切换，逻辑一致。
- **每帧**：多次物理子步后在主机侧更新一次粒子显示数据；绘图对存活粒子数量 **抽样上限**，减轻 PCIe 与 Matplotlib 压力。

### 2.4 可视化与交互

- 黑洞几何（视界环、吸积盘示意）仅在 **重置或参数变更** 时更新，避免每帧重绘静态要素。
- 粒子使用 **`Line2D` + 像素点 marker**（`,`）而非 `scatter`，以降低 PathCollection 刷新开销。
- 双列 **Matplotlib Slider**，旁附中文参数说明。

## 3. 本地 3D 程序原理（`fluid_sim_3d.py`）

### 3.1 数据结构

- 粒子位置：`pos (N, 3)`，`float32`，GPU 数组
- 粒子速度：`vel (N, 3)`，`float32`，GPU 数组
- 障碍体：当前版本已移除内部碰撞体，仅保留外层空间容器

### 3.2 物理更新流程

每帧步骤（GPU）：

1. 鼠标力场（吸引/排斥 + 旋转扰动）
2. 冲击波脉冲（点击触发）
3. 噪声扰动与底部对流
4. 重力与阻尼
5. 速度裁剪（限速）
6. 外层球形容器碰撞反射

### 3.3 GPU 优化策略

- 尽量使用 `cupy` 向量化，避免 Python 循环
- 减少 `cp.any()` 等同步点造成的 CPU-GPU 栅栏
- 渲染采用 **GPU 光栅化优先**：
  - 在 GPU 上完成投影/颜色/点写入
  - 每帧只回传最终画布，而非全量粒子中间态
- GPU 监控采样降频，降低 CPU 轮询开销

### 3.4 交互系统

- 左键：吸引
- 右键：排斥
- 中键：暂停/恢复（双击重置）
- 滚轮：调力
- UI 滑块：粒子数量目标值（拖动预览，松开生效）

## 4. Web 预览链原理（`app.py` + `web/main.js`）

### 4.1 通信模型

- 协议：WebSocket（双向）
- 前端 -> 后端：鼠标输入/参数变更
- 后端 -> 前端：粒子数据流（压缩二进制）

### 4.2 后端计算

- 默认优先 CUDA（`CuPy`）
- 不可用时回退 CPU（`NumPy`）
- 通过采样与量化控制带宽和抖动

### 4.3 数据压缩

为降低传输压力，后端将坐标量化为 `uint16`：

- Header: `count + sim_width + sim_height`
- Body: `uint16` 坐标流

前端按当前画布尺寸解码还原并渲染。

---

## 5. 关键性能瓶颈与处理

### 瓶颈 A：CPU-GPU 同步过多

- 现象：帧率抖动、交互滞后
- 处理：减少同步点，尽量在 GPU 完成整段计算

### 瓶颈 B：WebSocket 推流过大

- 现象：`keepalive ping failed`、前端卡顿
- 处理：降采样、降帧、二进制压缩、异常帧校验

### 瓶颈 C：浏览器渲染链不稳定

- 现象：白屏或掉帧
- 处理：上下文降级、渲染 try/catch、模式降级策略

---

## 6. 为什么“本地 3D”为核心

浏览器端受限于安全沙箱和渲染管线，不适合承担极限 CUDA 场景。  
因此项目定位为：

- **本地 3D = 核心体验**
- **Web = 预览与联调**
